# 처리율 제한 장치 프로젝트 시나리오

---

1. 프로젝트 배경

운영 중인 커뮤니티 서비스에서 특정 게시글이나 인기 API에 짧은 시간 동안 요청이 몰리는 상황이 발생한다.
일부는 정상 사용자 유입이지만, 일부는 새로고침 반복이나 비정상 호출로 인해 서버 자원을 과도하게 사용한다.

이로 인해 다음 문제가 생긴다.
• 애플리케이션 서버 부하 증가
• DB 조회량 급증
• 특정 사용자의 과도한 호출로 전체 사용자 경험 저하
• 로그인/비로그인 사용자 간 공정하지 않은 사용량 분배

그래서 API 요청 수를 일정 기준으로 제한하는 처리율 제한 장치를 설계하고 구현한다.

⸻

2. 서비스 가정

서비스는 게시글 조회 기능이 있는 커뮤니티 서비스라고 가정한다.

대상 API
• GET /posts/{postId}

API 특징
• 조회 요청이 매우 많다
• 인기 게시글에 트래픽이 몰릴 수 있다
• 비로그인 사용자는 IP 기준으로 식별해야 한다
• 로그인 사용자는 사용자 ID 기준으로 식별할 수 있다

⸻

3. 목표

이 프로젝트의 목표는 단순히 요청을 막는 것이 아니라,
현실적인 제약이 있는 환경에서 rate limiter를 점진적으로 설계·구현·개선하는 것이다.

최종적으로 다음을 보여주는 것을 목표로 한다.
• 단일 서버 환경에서의 기본 처리율 제한 구현
• 동시성 문제 재현 및 개선
• 다중 서버 환경에서의 한계 분석
• Redis 기반 분산 처리율 제한 장치 구현
• 알고리즘별 차이점 비교
• 부하 테스트를 통한 검증

⸻

4. 요구사항

기능 요구사항
• 비로그인 사용자는 IP 기준으로 요청 횟수를 제한한다
• 로그인 사용자는 사용자 ID 기준으로 요청 횟수를 제한한다
• 관리자는 제한 대상에서 제외한다
• 제한 초과 시 429 Too Many Requests 를 반환한다
• 응답 헤더에 남은 요청 수와 제한 정보를 포함한다

정책 요구사항
• 비로그인 사용자: 1분에 30회
• 관리자: 제한 없음

비기능 요구사항
• 동시 요청 상황에서도 가능한 한 정확해야 한다
• 서버가 여러 대여도 동일한 제한 정책이 적용되어야 한다
• 추후 다른 API에도 확장 가능해야 한다
• 알고리즘 교체가 가능하도록 구조를 분리해야 한다
⸻

6. 식별 기준

요청자를 아래 우선순위로 식별한다.

1순위: 로그인 사용자
• userId

2순위: 비로그인 사용자
• clientIp

즉 로그인한 경우에는 IP가 아니라 사용자 ID 기준으로 제한한다.
같은 IP를 여러 사용자가 공유할 수 있기 때문에, 로그인 사용자는 사용자 단위로 보는 것이 더 현실적이다.

⸻

7. 1차 시나리오: 단일 서버 + 메모리 기반 Fixed Window

가장 먼저 단순한 방식으로 시작한다.

동작 방식
• 사용자별 key를 만든다
• 현재 1분 윈도우 내 요청 수를 저장한다
• 제한 횟수를 넘으면 차단한다

예시
• user:123 -> 1분 동안 100회
• ip:192.168.0.1 -> 1분 동안 30회

기대 효과
• 빠르게 기본 rate limiting 동작을 구현할 수 있다
• 인터셉터/필터 수준에서 적용 가능하다

예상 한계
• 서버 재시작 시 카운트가 사라진다
• 서버가 여러 대면 인스턴스마다 카운트가 달라진다
• 동시성 상황에서 요청 수가 정확하지 않을 수 있다

⸻

8. 2차 시나리오: 동시성 문제 재현

단일 서버 메모리 기반 구현 후,
같은 사용자로 동시에 많은 요청을 보내 동시성 문제를 재현한다.

테스트 상황
• 로그인 사용자 1명이 동시에 200개 요청 전송
• 정책상 100개까지만 허용되어야 함

관찰 포인트
• 실제로 100개 초과 요청이 통과하는지
• count 증가가 원자적으로 처리되지 않아 오차가 발생하는지

개선 목표
• 단일 서버 내에서 동시성 제어를 적용한다
• Atomic 연산 또는 thread-safe 구조로 개선한다

⸻

9. 3차 시나리오: 다중 서버 환경 문제

이제 서버가 3대로 늘어난 상황을 가정한다.

환경
• Load Balancer
• API Server A / B / C
• 각 서버가 자체 메모리로 요청 수를 관리

문제

같은 사용자가 서버 A, B, C로 나뉘어 요청하면
서버마다 별도 카운트가 올라가므로 전체 제한이 깨진다.

예를 들어:
• 서버 A에서 40회 허용
• 서버 B에서 40회 허용
• 서버 C에서 40회 허용

결과적으로 100회 제한이어야 하는데 120회 이상 허용될 수 있다.

결론
• 메모리 기반 limiter는 분산 환경에서 신뢰할 수 없다
• 공유 저장소가 필요하다

⸻

10. 4차 시나리오: Redis 기반 Fixed Window

다중 서버 환경 문제를 해결하기 위해 Redis를 도입한다.

동작 방식
• 요청자별 key를 Redis에 저장
• 요청이 오면 Redis INCR 로 카운트 증가
• key에 TTL을 설정해 1분 후 만료

예시 key
• rate_limit:user:123:202603182330
• rate_limit:ip:10.0.0.1:202603182330

기대 효과
• 여러 서버가 동일한 카운터를 공유한다
• Redis의 원자 연산을 활용할 수 있다
• 수평 확장 환경에서도 정책이 일관되게 적용된다

남는 문제
• Fixed Window 특유의 경계값 문제가 있다

⸻

11. 5차 시나리오: Fixed Window 경계값 문제

Fixed Window는 구현이 단순하지만 경계에서 비정상적으로 많은 요청을 허용할 수 있다.

예시

제한: 1분 100회
• 12:00:59 에 100회 요청
• 12:01:00 에 100회 요청

실제로는 1초 사이에 200회가 처리될 수 있다.

의미
• 제한은 존재하지만 burst 트래픽 제어가 부정확하다
• 사용자 경험과 서버 보호 측면에서 더 정교한 방식이 필요하다

⸻

12. 6차 시나리오: Sliding Window으로 개선

이제 더 현실적인 알고리즘으로 발전시킨다.

Sliding Window

최근 60초 기준으로 요청 수를 계산한다.

장점
• Fixed Window보다 정확하다

단점
• 구현과 저장 비용이 더 크다

프로젝트 방향

최종 개선안은 Redis 기반 Sliding Window로 정한다.

⸻

13. 테스트 시나리오

구현 후에는 반드시 부하 테스트로 검증한다.

테스트 1

정상 요청
• 로그인 사용자 50회 요청
• 모두 허용되어야 한다

테스트 2

제한 초과 요청
• 로그인 사용자 120회 요청
• 100회까지만 허용, 나머지는 429

테스트 3

동시 요청
• 같은 사용자로 200개 동시 요청
• 허용 수가 제한값 근처에서 정확히 제어되는지 확인

테스트 4

다중 사용자
• 여러 사용자 동시 요청
• 각 사용자별 제한이 독립적으로 적용되는지 확인

테스트 5

다중 서버 환경
• 서버 A/B/C 에 요청 분산
• Redis 도입 전후 결과 비교

관찰 지표
• 허용 요청 수
• 차단 요청 수
• 평균 응답 시간
• Redis 명령 호출 수
• 알고리즘별 정확도 차이

⸻

14. 응답 정책

정상 응답 헤더 예시
• X-RateLimit-Limit
• X-RateLimit-Remaining
• X-RateLimit-Reset

차단 응답 예시

{
"code": "RATE_LIMIT_EXCEEDED",
"message": "요청 한도를 초과했습니다."
}

이렇게 하면 단순 차단이 아니라 API 사용자에게도 현재 상태를 알려줄 수 있다.

⸻

15. 최종적으로 보여주고 싶은 것

이 프로젝트를 통해 보여주려는 핵심은 아래다.
• 왜 rate limiter가 필요한가
• 단순 구현은 왜 분산 환경에서 깨지는가
• 동시성 문제는 어떻게 드러나는가
• Redis를 왜 쓰는가
• Fixed Window의 한계는 무엇인가
• Sliding Window는 뭔가?

⸻

한 줄로 정리한 최종 시나리오

“인기 게시글 조회 API에 요청이 몰리는 커뮤니티 서비스에서, 로그인/비로그인 사용자별로 다른 제한 정책을 적용하는 처리율 제한 장치를 설계하고, 메모리 기반 구현에서 시작해 Redis 기반 분산 환경 대응 및
알고리즘 개선까지 진행한다.”
